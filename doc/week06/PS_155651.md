# 프로그래머스 호텔 대실

[문제 링크](https://school.programmers.co.kr/learn/courses/30/lessons/155651)

## 초기 접근
- 입실 시간과 퇴실 시간을 가진 예약 클래스 생성
- 주어진 예약 시간 데이터를 `ArrayList`로 받아서 시작 시간 순으로 정렬
- `ArrayList`로 방 자료구조 생성 후 첫 번째 예약 넣음
- 두 번째 예약부터 방에 들어있는 예약의 종료시간과 넣을 예약의 시작 시간을 비교하여
- 시작 시간이 종료 시간보다 빠르면 방에 예약 추가 (방을 하나 더 추가)
- 그렇지 않으면 해당 자리에 새 예약으로 대체
- 예약을 모두 받은 뒤 방 자료구조의 크기가 방 수가 된다.

### 초기 코드
```
import java.time.LocalTime;
import java.util.ArrayList;

public class Solution {
	
	static class Book {
		LocalTime start;
		LocalTime end;
		
		public Book(LocalTime start, LocalTime end) {
			this.start = start;
			this.end = end;
		} //Constructor
	} //class - Book
	
	public int solution(String[][] book_time) {
        int answer = 0;
        ArrayList<Book> books = new ArrayList<>();
        
        for (String[] s : book_time) {
        	LocalTime start = LocalTime.parse(s[0]);
        	LocalTime end = LocalTime.parse(s[1]);
        	
        	books.add(new Book(start, end));
        } //for - insert books
        books.sort((o1, o2) -> o1.start.compareTo(o2.start));
        
        ArrayList<Book> rooms = new ArrayList<>();
        rooms.add(books.get(0));
        
        for (int i = 1; i < books.size(); i++) {
        	 Book cur = books.get(i);
        	 
        	 for (int j = 0; j < rooms.size(); j++) {
        		 if (cur.start.isAfter(rooms.get(j).end.plusMinutes(10))) {
        			 rooms.set(j, cur);
        		 } else {
        			 rooms.add(cur);
        		 } //if ~ else
        	 } //for - b
        } //for - i
        answer = rooms.size();
        
        return answer;
    }
}
```

### 문제점
- 현재 코드는 구조적으로 현재 예약을 새 방에 무한대로 넣고 있는 구조라 메모리 초과 발생
- 단순 비교 방식으로 접근하여 로직이 비효율적

## 개선 방향
- 종료 시간과 시작 시간을 단순 비교하는 방식이 아닌 종료 시간 기준으로 관리해야 함
- 현재 시간 기준으로 가장 빨리 끝나는 방을 찾기 위해 우선순위 큐를 사용하였고, 종료 시간만 넣음
- 가장 빨리 종료되는 방이 사용 가능하면 재사용, 아니면 새로운 방 추가

### 개선 코드
```
import java.time.LocalTime;
import java.util.ArrayList;
import java.util.PriorityQueue;

class Solution {
	
	static class Book {
		LocalTime start;
		LocalTime end;
		
		public Book(LocalTime start, LocalTime end) {
			this.start = start;
			this.end = end;
		} //Constructor
	} //class - Book
	
    public int solution(String[][] book_time) {
        int answer = 0;
        ArrayList<Book> books = new ArrayList<>();
        
        for (String[] s : book_time) {
        	LocalTime start = LocalTime.parse(s[0]);
        	LocalTime end = LocalTime.parse(s[1]);
        	
        	books.add(new Book(start, end));
        } //for - insert books
        books.sort((o1, o2) -> o1.start.compareTo(o2.start));
        
        PriorityQueue<LocalTime> pq = new PriorityQueue<>();
        
        for (Book b : books) {
        	if (!pq.isEmpty() &&
        			!pq.peek().plusMinutes(10).isAfter(b.start)) {
        		pq.poll();
        	} //if
        	
        	pq.add(b.end);
        } //for - b
        answer = pq.size();
        
        return answer;
    } //Solution
}
```

### 회고
- 리스트로 직접 관리하려 하면 상태가 복잡해진다는 것을 느꼈다.
- 이런 문제는 전체 관리가 아니라 최소 후보만 관리하는 구조가 핵심이었다.
- 우선순위 큐는 이번 처럼 **현재 가장 유리한 선택 하나만 뽑는 구조**에 적합하다는 것을 배웠다.
- `LocalTime` 자료형의 시간을 비교하는 메소드 `isAfter()`, `isBefore()`을 처음 사용해 보았다.
  이 두 메소드는 같은 시간일 경우에는 `false`를 반환하기 때문에 이번 문제에서 종료 시간이 시작 시간보다 빠르거나 같다는 조건을 얻기 위해 `!isAfter()`를 사용하였다.
- `Localtime` 자료형에서 시간을 더하는 메소드 `plusMinutes()`를 알게 되었다. 이외에도 시간, 초, 나노초 등을 더하는 메소드도 있다.
